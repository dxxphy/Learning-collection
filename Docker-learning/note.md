Docker 核心命令与原理速查笔记

Docker 的命令按照“镜像（模板） -> 容器（实例） -> 排错 -> 清理”的逻辑分类，掌握这些可以覆盖 95% 以上的日常开发和部署场景。

1. 核心原理：Docker 的文件系统层级

在理解命令前，必须先理清 Docker 容器内部的文件结构：

1.1 容器内部就是一个独立的 Linux

容器内部的文件层级，本质上就是一个标准的 Linux 文件系统（FHS）。
你可以把容器内部当成一台全新的 Linux 电脑。当你进入容器执行 ls / 时，会看到：

/bin 和 /sbin：存放基础命令（如 ls, sh）。

/etc：存放系统和软件的配置文件。

/var：存放经常变化的文件（如日志、数据库文件）。

/usr：存放用户安装的应用程序和文件。

/mnt：Mount（挂载）的缩写，系统预留的空目录，常用于临时挂载外部文件。

1.2 分层结构（UnionFS）的魔力

镜像层（只读）： 镜像是由多层叠加而成的只读模板。底层的操作系统、中间安装的软件，每一层都是不可修改的。

容器读写层： 当容器启动时，Docker 会在只读的镜像层之上，盖一层“读写层”。你在容器内部新建文件、修改配置，都只发生在最上面的读写层。一旦容器被 docker rm 删除，这个读写层就会随之销毁，数据灰飞烟灭。

2. 宿主机与容器的目录关系：Volume 挂载 (-v)

为了解决容器删除后数据丢失，以及在开发时频繁向容器内部拷贝代码的痛点，Docker 提供了**挂载（Volume / Bind Mount）**机制。

2.1 什么是挂载？

挂载相当于在宿主机（你的物理电脑）和容器的平行宇宙之间“打了一个虫洞”。通过启动命令中的 -v 参数，你可以将宿主机的某个真实目录，强行映射到容器内部的某个目录上。

2.2 语法解析：-v 宿主机路径:容器内部路径

例如：docker run -v /home/user/my_code:/mnt/dev my_image

逻辑： 将你物理电脑上的 /home/user/my_code 目录，直接穿透并覆盖到容器内部的 /mnt/dev 目录下。

效果： 1. 容器内的程序读写 /mnt/dev 时，实际上就是在实时读写你物理电脑上的 /home/user/my_code。
2. 哪怕容器被删除了，你在该目录下产生的文件，依然安全地保存在物理机的 /home/user/my_code 中。

注意点： 挂载是“覆盖”关系。如果你把宿主机的一个空目录，挂载到了容器内部一个原本有文件的关键系统目录上，会导致容器内部该目录被“变相清空”，从而引发服务启动失败。

3. 用户权限管理 (避免频繁 sudo)

默认情况下，Docker 守护进程绑定的是 Unix Socket，这个 Socket 的所有者是 root，因此普通用户执行 Docker 命令时必须加上 sudo（例如 sudo docker ps）。为了开发方便，我们可以将当前用户加入 docker 用户组。

3.1 告别 sudo 的配置流程

创建 docker 用户组（如果尚未存在）：

sudo groupadd docker


将当前用户加入 docker 组：

sudo usermod -aG docker $USER


(注：-aG 代表 Append to Group，追加到用户组)

使组权限更改立即生效：

newgrp docker


原理解释： 正常情况下，usermod 修改用户组后，需要用户注销并重新登录才能生效。newgrp docker 命令的作用是在当前终端会话中，临时将当前用户的有效用户组切换为 docker，从而立刻获得操作 Docker 的权限，免去了重启或注销的麻烦。

验证（现在可以直接运行了）：

docker run hello-world


提示：配置完成后，建议关闭当前终端并重新打开，以确保所有新终端都能继承权限。

4. 镜像管理 (Image)

镜像是只读的模板，相当于“菜谱”或面向对象中的 Class。

拉取镜像

docker pull <镜像名>:<标签>
# 示例: docker pull mysql:8.0 (如果不加标签，默认拉取 latest 最新版)


查看本地镜像

docker images


删除本地镜像

docker rmi <镜像名或镜像ID>


构建自定义镜像

docker build -t <自定义镜像名>:<标签> .


5. 容器管理 (Container)

容器是镜像运行起来的实体，相当于“做好的菜”或面向对象中的 Object。

创建并运行容器 (最常用)

docker run [参数] <镜像名>


常用参数解析：

-d: 后台运行 (Detached mode)，不占用当前终端。

-p 主机端口:容器端口: 端口映射（例：-p 8080:80，访问本机的 8080 即访问容器的 80）。

-v 主机绝对路径:容器绝对路径: 目录挂载（实现数据持久化或代码同步）。

--name <名字>: 指定容器名称。

--rm: 容器停止后自动删除读写层，适合一次性测试。

查看容器状态

docker ps      # 仅列出正在运行的容器
docker ps -a   # 列出所有容器 (包括已停止的，排错必备)


启停与删除容器

docker start <容器名或ID>   
docker stop <容器名或ID>    
docker restart <容器名或ID> 
docker rm <容器名或ID>      # 删除已停止的容器


6. 进阶排错与运维 (Debugging)

排错“三板斧”。

查看容器日志 (排错首选)

docker logs -f <容器名或ID>


进入容器内部

docker exec -it <容器名或ID> /bin/bash
# 如果镜像是基于 Alpine 构建的，可能没有 bash，需改用 sh:
# docker exec -it <容器名或ID> sh


宿主机与容器互传文件 (不使用挂载时的临时方案)

docker cp <宿主机路径> <容器名>:<容器内路径>  # 从本机拷入容器
docker cp <容器名>:<容器内路径> <宿主机路径>  # 从容器拷出到本机


7. 系统级清理 (核弹操作)

一键清理无用资源

docker system prune


(清理已停止的容器、未使用的网络和悬空镜像。如果加 -a 参数，会额外删除所有未被当前运行容器使用的镜像，需谨慎！)

💡 核心学习法则

开发流程： build (打包) -> run (运行) -> logs (看运行状态) -> stop (停止) -> rm (删除)

排错三步曲： docker ps -a (看谁挂了) -> docker logs (看为什么挂) -> docker exec (进内部文件系统排查环境或路径问题)
